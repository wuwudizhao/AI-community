# Liftoff Validation MVP

## 1. 为什么缩小范围

Liftoff 的核心风险不是能否开发出完整论坛，而是用户是否愿意围绕 AI 赚钱机会、项目验证、变现过程与商业复盘完成真实发布、回复和回访。点赞、收藏、关注、AI 能力和 Hero 游戏会增加实现成本，却不能优先回答这一问题。

因此，从 Phase 2 起，项目从“完整论坛功能建设”切换为 Validation MVP：只建立验证核心用户行为链所需的最小产品，并用真实用户行为决定是否继续投资。

Phase 1 工程骨架全部保留。PostgreSQL 容器和 migration 的在线验证仍因当前机器缺少 Docker/PostgreSQL 而处于环境阻塞状态。

## 2. 验证假设

Validation MVP 验证以下假设：

1. 想利用 AI 做副业的普通用户愿意浏览赚钱机会、副业项目、项目验证、拆解、获客、收入案例和失败复盘；
2. 访客愿意通过邮箱注册成为用户；
3. 注册用户愿意发布真实机会、验证过程、项目、收入结果或失败复盘；
4. 其他用户愿意回复这些内容；
5. 回复通知能够促使发帖用户返回讨论；
6. 用户在首次访问后愿意于 7 日内再次访问。
7. Vibe Coding、Agent、RAG、MCP 和 FDE 内容能够服务明确的项目或商业目标，而不是形成脱离结果的技术循环。

核心验证链：

```text
访问首页 → 浏览帖子 → 邮箱注册 → 发帖 → 回复 → 通知 → 回访
```

内容价值链：

```text
赚钱机会 → 项目验证 → 项目拆解 → AI Coding/技术优化 → 获客与变现 → 商业复盘
```

## 3. MVP 功能

### 3.1 公开体验

- 社区首页；
- 浅色、深色和跟随系统主题；
- 简单分类导航；
- 帖子列表与帖子详情；
- 简单用户主页。

### 3.2 账户

- 邮箱注册；
- 邮箱登录；
- 退出；
- 当前用户；
- 简单个人资料。

Phase 3 已选择邮箱注册、一次性邮箱验证与 PostgreSQL 服务端 Session；敏感 Token 不存 localStorage。

### 3.3 内容与讨论

- 发布帖子；
- 编辑、删除自己的帖子；
- 一级评论；
- 回复评论；
- 基础回复通知。

### 3.4 最小治理与运行

- 举报；
- 管理员删除帖子和评论；
- 管理员封禁用户；
- 最小管理页面；
- 基础部署、日志和数据备份。

## 4. 延期功能

- Hero 像素小游戏；
- 点赞、收藏和关注；
- 私信、排行榜、积分和徽章；
- GitHub、Google 和微信登录；
- AI 自动回答、AI 推荐和 AI 搜索；
- RAG、Agent 和 Redis；
- 完整对象存储体系；
- 复杂富文本编辑器；
- 复杂板块层级与完整审核工作台；
- 高级 SEO 与原生 App。

延期不等于承诺后续一定开发。只有验证链成立且存在真实需求时，相关功能才重新进入评估。

## 5. 测试用户规模

完成部署和基本治理后，计划邀请 10–50 名目标用户进行小范围验证。该范围是验证计划，不是当前用户规模、增长承诺或上线目标。

招募顺序为：想利用 AI 做副业的人、独立开发者、AI 创业者、AI 工程师。工程师不再作为默认第一用户群。

## 6. 验证指标

以下指标均为待验证目标，当前没有真实数据，也不预设虚假的达标值：

- 访问到注册转化率；
- 注册用户发帖率；
- 帖子回复率；
- 发帖用户数量；
- 7 日回访率；
- 无回复帖子比例。
- 机会内容进入项目验证的比例；
- 发布后补充验证结果或商业复盘的比例；
- 收入案例对成本、周期、渠道和口径的说明完整度；
- 技术内容连接明确项目或变现目标的比例。

统计口径、观察窗口、基线和目标阈值需在 Phase 6 邀请测试前确认。

## 7. 继续投资条件

在小范围测试中同时出现以下信号时，才继续扩展：

- 有多名非项目成员完成真实发帖；
- 真实帖子持续获得有意义的回复；
- 回复通知能带来可观察的回访；
- 用户愿意在首次体验后再次发布或回复；
- 内容治理成本在现有团队可承受范围内；
- 用户明确提出且反复出现的功能需求能被验证。

具体数量阈值待 Phase 6 前确认。

## 8. 停止或调整条件

出现以下情况时，应停止扩功能并调整定位、用户来源或产品流程：

- 用户浏览但长期不愿注册；
- 注册用户不愿发布真实内容；
- 帖子长期无人回复；
- 用户只消费官方内容，无法形成用户间讨论；
- 通知不能带来回访；
- 低质量、广告或 AI 搬运内容使治理成本不可承受；
- 用户更需要项目展示、即时群聊或专业问答，而不是当前论坛形态。
- 用户只浏览“赚钱”标题，却不愿进行项目验证或提交真实结果；
- 收入案例缺乏可解释边界，导致社区信任或治理成本不可接受。

停止或调整不以已投入开发量为判断依据，而以核心验证链的真实证据为依据。

# Phase 3B 邮件验收状态

Resend 适配代码和受控 smoke 脚本已完成，但真实邮箱投递尚未执行。需要在安全环境注入密钥和已验证发件地址，运行 `scripts/mail-smoke.ps1`，人工确认收件、发件人、主题、链接、验证、登录、me、退出及重发行为后，才能将 Phase 3 标为 PASS。

# Phase 5 互动闭环状态

Phase 5 已以两个真实测试账户、PostgreSQL、HTTP 和浏览器完成“发帖 → 评论 → 通知 → 回复 → 通知 → 已读 → 删除占位 → 回访”闭环。站内通知不依赖邮件；Phase 3 的 Resend 在线阻塞仍独立存在并保持 PARTIAL。当前不实现实时推送、邮件通知、点赞、收藏、关注或私信。

# Beta 收口门禁

Web 质量门禁通过不替代基础设施与邮件验收。公开 Beta 必须同时满足 PostgreSQL 在线、migration 状态一致、真实邮件验证闭环、浏览器控制台无错误和无 Mock 数据。当前结论保持 PARTIAL。
